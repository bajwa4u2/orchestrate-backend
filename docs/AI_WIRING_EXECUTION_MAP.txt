ORCHESTRATE AI WIRING EXECUTION MAP FOR CODEX
=============================================

Status
------
Backend AI control layer is not yet fully wired into production execution. The current backend already contains AI, execution, worker, provider, source, reply, meeting, email, workflow, and policy zones. Codex must wire into these existing zones instead of creating duplicate systems.

Primary runtime ownership
-------------------------
- src/execution/execution.service.ts
  Owns dispatch loop, campaign continuity, queueing, runJob, immediate send, bootstrap/refill logic, reply classification, and meeting handoff paths.

- src/workers/workers.service.ts
  Owns registry of concrete workers and maps JobType to worker services.

- src/ai/ai.service.ts
  Existing AI-capable service. Contains campaign activation bootstrap, outbound draft generation, growth message/sequence generation, reminders, agreements, and statement summaries.

- src/workflows/workflows.service.ts
  Workflow run creation and lifecycle tracking.

- prisma/schema.prisma
  Authority for persisted reality. Must be checked before adding any new decision or evaluation records.

Phase 1: Lead to first outreach
-------------------------------
Approved choke points:
- src/campaigns/campaigns.service.ts activation/restart creates LEAD_IMPORT jobs.
- src/execution/execution.service.ts queueLeadSend, runImmediateSendForLead, runLeadImportBootstrap, maintainCampaignContinuity.
- src/workers/lead_import/lead-import.worker.service.ts imports/discovers/upserts leads and creates FIRST_SEND jobs.
- src/workers/message_generation/message-generation.worker.service.ts generates message content using AiService.
- src/workers/first_send/first-send.worker.service.ts performs actual first-send execution and follow-up queueing.

Decision to introduce:
- SEND_FIRST_OUTREACH
- WAIT
- STOP
- ESCALATE_OPERATOR

Hard blocks:
- missing email
- suppressed contact
- invalid campaign state
- deliverability/policy denial
- no active billing/subscription where campaign execution requires it
- missing agreement/representation authorization where required

Phase 3: Reply handling
-----------------------
Approved choke points:
- src/replies/replies.service.ts ingestInboundReply/processReply
- src/workers/reply_classification/reply-classification.worker.service.ts
- src/execution/execution.service.ts runReplyClassification

Decision outputs:
- INTERESTED -> meeting handoff
- NEGATIVE -> stop/suppress/finalize lead path
- NEUTRAL -> later follow-up or operator review
- OUT_OF_OFFICE -> delay

Required persistence:
- intent
- confidence
- reasoning
- chosen next action
- outcome later

Decision record requirement
---------------------------
Before any authoritative AI action is executed, the system must persist a decision record or equivalent DB-backed record. Existing models should be inspected first, especially ActivityEvent, AuditLog, Alert, PolicySetting, WorkflowRun, AdaptationDecision, QualificationDecision, ProviderUsageLog, SourceRun, Job, JobRun, OutreachMessage, Reply, and Meeting.

Do not add a new Prisma model until existing models are assessed and the user explicitly approves schema/migration work.
